Part 01 · Origins

Where this started

2010 — 2026

Before there were agents, before there were LLMs in production at all, there was a much older and more boring-sounding problem: understanding what a distributed system is doing right now, from the outside, without stopping it to look. That problem — and the tooling built to solve it — is the direct ancestor of almost everything used to watch an AI agent today. Knowing that lineage explains both the vocabulary everyone still uses (trace, span) and the current mess of competing tools — this field hasn't converged once yet, it's converged twice, and it's arguably in the middle of a third convergence right now.

1.1 — Distributed tracing becomes the industry standard (2010–2021)

In 2010, Google published the Dapper paper, describing a large-scale tracing system it had already been running internally for two years. Dapper itself was never released — but the paper fixed the vocabulary the entire field still uses: a trace is the end-to-end path of one request through a system, and a span is one timed, named unit of work inside it.

Two years later, Twitter open-sourced Zipkin, directly inspired by Dapper — the first tracing tool anyone outside Google could actually install and run. Uber followed with an internal project inspired by both Dapper and Zipkin, which it open-sourced as Jaeger. By late 2016, the Cloud Native Computing Foundation had accepted OpenTracing — a vendor-neutral API specification for tracing — as its third hosted project. OpenTracing deliberately shipped as an API only, with no implementation, so any vendor could plug in underneath it.

Google answered with a competing effort, OpenCensus, which took the opposite approach: a full implementation covering both traces and metrics, rather than just an API. Having two credible, incompatible standards recreated the exact problem standards exist to prevent — library authors had to pick one or maintain both, and switching observability vendors meant re-instrumenting the whole codebase.

In May 2019, the two projects did the sensible thing and merged into a single effort: OpenTelemetry. Its tracing specification reached version 1.0 stability in 2021. By this point the industry had settled on its "three pillars" — logs, metrics, and traces — later nicknamed Observability 1.0. This was a world built for watching microservices, not language models: every request has a status code, a duration, and (mostly) a deterministic path.

1.2 — The gap: agents break the old assumptions (2023)

The three pillars assume a request either succeeds or fails in a way a status code can capture. An LLM call breaks that assumption in the way that matters most: the same input can produce a different output every time, the "failure" is very often a confident 200-response with a wrong answer inside it, and the thing you actually need to debug — the exact prompt, the exact completion, the token cost — isn't a metric or a log line, it's a large, unstructured, expensive-to-store blob. Traditional APM could tell you latency and error rate. It had nothing to say about whether the model hallucinated.

2023 is the year the field answers this, four different ways at once, none of them standardized yet:

  • Langfuse and LangSmith both launch the same year — full platforms that treat the prompt/response/token/cost record as a first-class object, not something squeezed into a generic log line.
  • OpenInference, started by Arize, defines its own LLM-span attribute namespace — because OpenTelemetry doesn't have one yet, and someone has to.
  • Helicone takes a proxy-based bet instead: sit in front of the model API and log the wire traffic, no SDK instrumentation required.
  • OpenLLMetry (by Traceloop) takes a fourth bet: a vendor-neutral, OTel-based instrumentation SDK you can point at whichever backend you choose later.

Four different architectural answers to the same new problem, arriving in the same twelve months, all incompatible with each other.

1.3 — Standardization begins (April 2024 – January 2025)

In April 2024, OpenTelemetry formed a dedicated GenAI Special Interest Group under its Semantic Conventions SIG. The mandate: agree on one shared vocabulary — attribute names for the model used, token counts, latency — so that instrumentation written by different vendors produces telemetry that means the same thing everywhere, instead of five slightly different dialects of the same idea.

In January 2025, Traceloop folded OpenLLMetry's semantic conventions directly into OpenTelemetry itself — turning a vendor-specific effort into part of the open standard rather than a competitor to it. In between those two dates, in November 2024, Anthropic released the Model Context Protocol, standardizing how agents call tools — which quietly created a whole new common span type (the tool call) that observability now has to describe just as carefully as the LLM call itself.

1.4 — Where it's landed: convergence and agent-specific extension (2025–2026)

Late in 2024, Charity Majors gave the shift a name — Observability 2.0: a move away from pre-aggregated dashboards you had to guess the shape of in advance, toward being able to ask any question of raw, structured event data after the fact. For agents this isn't just philosophy — you genuinely don't know in advance which failure mode you'll need to dig into.

Through 2025, Datadog launched AI Agent Monitoring at its DASH conference, adopting the OTel GenAI attributes natively, while Langfuse and LangSmith kept growing along different bets — Langfuse doubling down on OTel-native, self-hostable infrastructure, LangSmith doubling down on deep LangGraph integration. Through 2025 and into 2026, the GenAI SIG has pushed past single LLM calls into two new, still-experimental efforts: AI Agent Application Conventions (tasks, actions, state transitions, memory for multi-agent systems) and AI Agent Framework Conventions (adapters for CrewAI, AutoGen, LangGraph, Semantic Kernel, and others).

By 2026, OpenInference and OpenTelemetry's own gen_ai.* namespace are actively converging — most instrumentation now emits both attribute sets for compatibility during the transition. And the market is consolidating even as it keeps forking: Langfuse joined ClickHouse (the database that already powered its backend) in January 2026, while newer, agent-native entrants like Laminar argue the first generation of tools got the data model wrong for agent-shaped traffic in the first place — a single agent run can be 40 to 75 spans of nested tool calls, not the one prompt/response pair the 2023-era tools were designed around.

That's the whole arc: a 2010-era answer to "what is my microservice doing" became, through a fairly traceable chain of forks, mergers, and one deliberate standards effort, a 2026-era answer to "what is my agent doing" — using the same core vocabulary (trace, span), stretched to cover a problem it wasn't built for, plus one genuinely new layer bolted on top, because agents fail in ways a status code was never going to catch.

Part 02 · Foundations

Core concepts

What you're actually building, precisely defined

2.1 — The vocabulary

  • Trace — everything that happened during one agent run, start to finish. The top-level container.
  • Span — one timed unit of work inside a trace: one LLM call, one tool call, one retrieval step. Spans nest — a span can have child spans.
  • Generation — a specialized span for an LLM call specifically, carrying the exact prompt, the exact completion, model parameters, token counts, and cost — not just a start and end time.
  • Session — a group of traces belonging to the same conversation or user, useful once an agent has multi-turn memory rather than single isolated runs.
  • Observation — the generic term some platforms use for "anything inside a trace": a span, a generation, or a point-in-time event, treated as one interchangeable unit.

2.2 — The four layers (traces are the foundation, not the whole story)

Classic observability's three pillars still apply, but agents need a fourth layer stacked on top — this is the single biggest structural difference from traditional software observability.

Part 03 · Current state

The tooling landscape, mid-2026

Two separate decisions, not one

"Which observability tool" is actually two separate decisions bundled together: how do you emit the telemetry (the instrumentation layer), and where does it live and get queried (the backend/platform layer)? Keeping these separate is what protects you from re-instrumenting an entire codebase if you switch platforms later.

3.1 — Instrumentation layer

  • OpenTelemetry GenAI semantic conventions — the shared vocabulary itself (gen_ai.* attributes). Not a tool — a spec.
  • OpenLLMetry (Traceloop) — an OTel-native SDK implementing that vocabulary, vendor-neutral, ships to whichever backend you pick.
  • OpenInference (Arize) — the earlier, parallel namespace, now converging with OTel's own.

3.3 — A narrowing heuristic (not a recommendation)

  • Deep in LangGraph already, comfortable with a commercial tool → LangSmith.
  • Self-hosting or data residency is a hard requirement → Langfuse or MLflow, self-hosted.
  • The open question is retrieval quality / hallucination, not agent structure → Phoenix.
  • You want the fastest possible integration and don't need deep agent visibility yet → Helicone.
  • Already running Prometheus/Loki/Tempo and don't want a second platform → OpenObserve.
  • None of the above — you just want the safest, most-adopted neutral default → Langfuse is the closest thing to one in 2026.
Part 04 · Practice

Best practices

What the field has converged on so far

4.1 — Instrument at the right boundaries

Every external tool call (API, database, MCP server, search) gets its own span: name, redacted inputs, latency, status, error if any. The agent's own reasoning or planning step gets a span too, if the framework exposes it — that's what lets you answer "why did it pick that tool" after the fact. One root trace per agent run, always — never let a multi-step agent's work land as a flat list of unconnected events.

4.2 — Follow the semantic conventions; don't invent your own

Using gen_ai.* attribute names (or your platform's direct equivalent) keeps traces portable if you ever switch backend. Rolling your own attribute names is the single most common reason teams get stuck on one vendor.

4.3 — Redact before you export, not after

Prompts and tool payloads often carry secrets or personal data. Redact at the instrumentation layer. Once something has landed in a third-party backend, removing it is a much bigger job than never sending it.

4.4 — Propagate context across the whole trace

Attributes like user ID or session ID need to be attached to every span in a trace, not only the root span, or you can't filter and aggregate by them later. OpenTelemetry Baggage is the standard mechanism for this — set it once, and it copies to every span in the trace context automatically.

4.5 — Keep emit, store, and judge as three separate jobs

Don't let the instrumentation SDK, the storage backend, and the evaluation logic become one tangled thing. Keep the OTel/OpenLLMetry emission layer swappable, keep the backend swappable, and add evaluation as a layer on top once real traces exist to test it against — in that order.

4.6 — Metrics are a view over traces, not a separate thing to log twice

Compute latency percentiles, cost per run, and tool success rate from the trace data itself. Logging the same numbers twice, in two different systems, is how dashboards and traces quietly drift apart from each other.

4.7 — Sample deliberately once volume grows

Rule of thumb

At low volume, trace everything. Once an agent is handling real production traffic, decide a sampling strategy on purpose — always keep traces with errors or low eval scores, sample the rest — rather than accidentally dropping the interesting 1% along with the boring 99%.

Part 05 · Execution

How to build it, step by step

A generic sequence — works for any agent, any framework

  1. 01Pick the instrumentation SDK first, not the backend. OpenLLMetry, or your framework's built-in OTel support — this keeps you unlocked from any specific backend before you've even chosen one.
  2. 02Wrap every tool call and LLM call in a span at the point it happens in code — usually a decorator or context manager, rarely a rewrite.
  3. 03Give every agent run one root trace, and propagate session/user attributes as OTel Baggage so they land on every child span, not just the first one.
  4. 04Export via OTLP to the backend you chose in Part 03.
  5. 05Build three or four dashboards from the trace data: latency percentiles, cost per run, tool success rate, retry/loop count. Resist building more than that at first.
  6. 06Add evaluation last — attach an LLM-as-judge or rule-based score once real production traces exist to validate the judge against, not before.
  7. 07Close the loop — use eval scores to find the traces worth turning into regression test cases, so the next prompt or model change gets checked against real past failures.
Part 06 · Outlook

What's still unsettled

Mid-2026, and probably true for another year or two

Still experimental

Agent-specific OTel conventions — multi-agent tasks, actions, handoffs, memory — are still experimental as of mid-2026. Expect the attribute names in this exact space to keep changing.

Two standards, converging but not merged

MCP-level tracing conventions are being worked out alongside the LLM-call conventions, since tool calls through MCP are now as common a span type as the LLM call itself — and OpenInference and OTel's own namespace are still mid-convergence, not fully unified.

The market is forking, not settling

Langfuse consolidating into ClickHouse (Jan 2026) and agent-native entrants like Laminar arguing the first generation's data model doesn't fit agent-shaped traffic are both happening at once — expect more architectural forks like this over the next year, not fewer.

The practical takeaway from all of this: build on the open standard (OTel GenAI conventions) rather than a single vendor's proprietary attribute set, and keep raw traces queryable rather than only charting the dashboards you thought to build in advance. That's the whole idea behind Observability 2.0 — you rarely know ahead of time which question you'll need to ask.

Glossary

trace
Everything that happened during one agent run, start to finish.
span
One timed, named unit of work inside a trace — an LLM call, a tool call, a retrieval step. Spans nest.
generation
A span specialized for an LLM call — carries prompt, completion, model params, tokens, and cost.
session
A group of traces belonging to the same conversation or user.
OTLP
OpenTelemetry Protocol — the vendor-neutral wire format traces are exported in.
semantic convention
An agreed-on, standard name and meaning for a telemetry attribute (e.g. gen_ai.request.model), so different tools' output means the same thing.
sampling
Deliberately keeping only a subset of traces (e.g. all errors + a random sample of the rest) once volume makes tracing everything impractical.
LLM-as-judge
Using a separate model call to score whether a trace's output met some quality bar, instead of (or alongside) a human or rule-based check.
guardrail
An evaluation check placed in the critical path of a live request — able to block, retry, or fall back — rather than only run after the fact for monitoring.
