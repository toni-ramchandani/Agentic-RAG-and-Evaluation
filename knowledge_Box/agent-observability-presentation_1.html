One lineage: 2010 → 2026

Part 01 · Origins

The short version

2010–2021 — tracing becomes standard

Dapper → Zipkin → OpenTracing / OpenCensus → merge into OpenTelemetry → v1.0. Logs, metrics, and traces become "the three pillars" of software observability.

2023 — agents outgrow the pillars

Langfuse & LangSmith both launch. OpenInference (Arize) fills the OTel gap. Helicone and OpenLLMetry emerge — four different architectural bets, the same year.

2024–2026 — standardize, then extend

OTel forms the GenAI SIG (Apr '24) · MCP ships (Nov '24) · OpenLLMetry merges into OTel (Jan '25) · Datadog AI Agent Monitoring + "Observability 2.0" (2025) · agent-specific conventions + market consolidation (2026).

Part 02d · The concrete upgrade path

Two things worth doing next

Swap the export, not the instrumentation

ExecutionStep is already span-shaped. Getting to Langfuse / MLflow / OTel doesn't mean re-instrumenting — it means replacing execution.save()'s local JSON write with an OTLP export call. Same six step names, same boundaries.

One real blind spot: the Ollama router

OllamaProvider._route_action() makes its own LLM call to classify intent — answer_directly / create_support_ticket / ask_clarification — but it's nested inside provider_generate_response, not its own step. A slow or wrong routing decision is currently invisible in your traces. Concretely: give it its own execution.start_step("intent_routing", …).

Part 03 · Current state

The landscape, mid-2026

Langfuse

OTel-native, self-hostable — tracing, evals, and prompt management in one place.

MLflow Tracing

Apache 2.0, auto-instruments 60+ frameworks. Best if already ML/Databricks-adjacent.

Arize Phoenix

Strongest RAG-specific eval methodology — grounding and hallucination detection.

LangSmith

Deepest LangGraph / LangChain integration; commercial, best-in-class agent trajectory views.

Helicone

Proxy-based — fastest possible setup, weakest agent-structure visibility.

OpenObserve · Laminar

Unified telemetry store · agent-native pricing model for high-span-count traces.

Part 03 · Current state

Picking fast

  • → Deep in LangGraph already? LangSmith
  • → Self-hosting / data residency required? Langfuse or MLflow
  • → Question is hallucination, not structure? Phoenix
  • → Need the fastest possible integration? Helicone
  • → No strong preference either way? Langfuse — the safest neutral default
Part 04 · Practice

Best practices

  • Instrument at the real boundaries — every tool call gets its own spane.g. each entry in your TOOL_REGISTRY (tools.py)
  • Use gen_ai.* conventions — not your own attribute names
  • Redact before export — not after it's already in a backend
  • Propagate context via Baggage — on every span, not just the root
  • Keep emit / store / judge separate — three swappable jobs
  • Metrics = a view over traces — not a second thing to log
  • Sample deliberately — once real production volume arrives

The instrumentation pipeline

Backend-agnostic until step 3 — that's the whole point.

Part 05 · Execution

Seven steps

  1. 01Pick the instrumentation SDK first, not the backend e.g. OpenLLMetry, or Langfuse's @observe() decorator
  2. 02Wrap every tool call and LLM call in a span e.g. this repo's tool_execution step, around create_support_ticket()
  3. 03One root trace per run — propagate context as Baggage
  4. 04Export via OTLP
  5. 05Build 3–4 dashboards, no more
  6. 06Add evaluation last, once real traces exist
  7. 07Close the loop — regression cases from real failures
Part 06 · Outlook

What's still unsettled

Still experimental

Agent-specific OTel conventions — multi-agent tasks, handoffs, memory — are still experimental as of mid-2026.

Converging, not merged

MCP-level tracing conventions and the OpenInference / OTel namespaces are mid-convergence, not fully unified yet.

Forking, not settling

Langfuse joining ClickHouse and agent-native entrants like Laminar contesting the trace data model — both happening at once.
